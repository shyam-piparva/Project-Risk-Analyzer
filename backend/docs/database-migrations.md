# Database Migrations

This document describes the database migration system for the AI Project Risk Analyzer.

## Overview

The project uses [node-pg-migrate](https://github.com/salsita/node-pg-migrate) for managing database schema changes. This provides version control for the database schema with the ability to migrate up (apply changes) and down (rollback changes).

## Migration Files

Migrations are located in `backend/migrations/` and are executed in order based on their timestamp prefix:

1. **1700000000000_create-users-table.js** - Creates users table with authentication fields
2. **1700000000001_create-projects-table.js** - Creates projects table with constraints
3. **1700000000002_create-risk-analyses-table.js** - Creates risk_analyses table
4. **1700000000003_create-risks-table.js** - Creates risks table with scoring fields
5. **1700000000004_create-mitigations-table.js** - Creates mitigations table
6. **1700000000005_create-reports-table.js** - Creates reports table

## Database Schema

### Tables

- **users** - User accounts with authentication (email, password_hash, name, etc.)
- **projects** - Project details with team composition and technology stack (JSONB)
- **risk_analyses** - Risk analysis results for projects with overall scores
- **risks** - Individual risks identified in analyses with scores and categories
- **mitigations** - Mitigation strategies for risks with priority and implementation status
- **reports** - Generated PDF/CSV reports with metadata

### Key Features

- **UUID Primary Keys** - All tables use UUIDs generated by `uuid_generate_v4()`
- **Foreign Key Constraints** - Proper referential integrity with CASCADE deletes
- **Check Constraints** - Data validation at database level:
  - Valid date ranges (end_date > start_date)
  - Positive budgets and team sizes
  - Score ranges (0-100 for risks, 0-1 for probability/impact)
  - Valid enums for categories, statuses, priorities, and report types
- **Indexes** - Performance optimization on frequently queried fields:
  - User email lookups
  - Project and risk queries by user/project
  - Risk sorting by score (descending)
  - Filtering by category and status
- **Triggers** - Automatic `updated_at` timestamp updates for users and projects

## Running Migrations

### Prerequisites

Ensure PostgreSQL is running and set up your database connection:

```bash
# Option 1: Use DATABASE_URL environment variable
export DATABASE_URL="postgresql://postgres:postgres@localhost:5432/risk_analyzer"

# Option 2: Use individual environment variables (defined in .env)
DB_HOST=localhost
DB_PORT=5432
DB_NAME=risk_analyzer
DB_USER=postgres
DB_PASSWORD=postgres
```

### Commands

```bash
# Run all pending migrations
npm run migrate

# Rollback the last migration
npm run migrate:down

# Redo the last migration (down then up)
npm run migrate:redo

# Create a new migration file
npm run migrate:create my-migration-name
```

### Using Docker

When using Docker Compose, start the PostgreSQL service:

```bash
# Start PostgreSQL
docker-compose up -d postgres

# Wait for PostgreSQL to be ready, then run migrations
npm run migrate --workspace=backend
```

## Migration Structure

Each migration file exports two functions:

```javascript
/* eslint-disable camelcase */

exports.shorthands = undefined;

exports.up = (pgm) => {
  // Apply the migration (create tables, add columns, etc.)
  pgm.createTable('my_table', {
    id: { type: 'uuid', primaryKey: true, default: pgm.func('uuid_generate_v4()') },
    name: { type: 'varchar(255)', notNull: true },
  });
};

exports.down = (pgm) => {
  // Revert the migration (drop tables, remove columns, etc.)
  pgm.dropTable('my_table', { ifExists: true, cascade: true });
};
```

## Configuration

Migration settings are configured in `backend/.node-pg-migrate.json`:

```json
{
  "databaseUrl": { "env": "DATABASE_URL" },
  "migrationsTable": "pgmigrations",
  "dir": "migrations",
  "direction": "up",
  "schema": "public",
  "decamelize": true,
  "checkOrder": true,
  "verbose": true
}
```

## Verification

After running migrations, verify the schema:

```bash
# List all tables
docker exec risk-analyzer-db psql -U postgres -d risk_analyzer -c "\dt"

# List all indexes
docker exec risk-analyzer-db psql -U postgres -d risk_analyzer -c "\di"

# Describe a specific table
docker exec risk-analyzer-db psql -U postgres -d risk_analyzer -c "\d projects"

# Check migration history
docker exec risk-analyzer-db psql -U postgres -d risk_analyzer -c "SELECT * FROM pgmigrations;"
```

## Best Practices

1. **Never modify existing migrations** - Create new migrations for schema changes
2. **Always provide down() functions** - Enable rollback capability
3. **Test migrations locally** - Before deploying to production
4. **Use transactions** - Migrations run in transactions by default (atomic)
5. **Keep migrations small** - One logical change per migration
6. **Use meaningful names** - Describe what the migration does

## Troubleshooting

### Migration fails with "relation already exists"

The migration has already been run. Check the migration history:

```bash
docker exec risk-analyzer-db psql -U postgres -d risk_analyzer -c "SELECT * FROM pgmigrations;"
```

### Need to reset the database

```bash
# Drop and recreate the database
docker exec risk-analyzer-db psql -U postgres -c "DROP DATABASE risk_analyzer;"
docker exec risk-analyzer-db psql -U postgres -c "CREATE DATABASE risk_analyzer;"

# Run migrations again
npm run migrate --workspace=backend
```

### Migration stuck or incomplete

Check the `pgmigrations` table and manually remove the incomplete migration record if needed:

```sql
DELETE FROM pgmigrations WHERE name = 'problematic-migration-name';
```

Then run migrations again.

## Schema Diagram

```
users
  ├── projects (user_id FK)
  │   ├── risk_analyses (project_id FK)
  │   │   └── risks (analysis_id FK)
  │   │       └── mitigations (risk_id FK)
  │   └── reports (project_id FK, analysis_id FK, generated_by FK)
```

All foreign keys use `ON DELETE CASCADE` to maintain referential integrity.
